### 2.5 polish 后复扫 forbidden_items（必做 · Round 18.3）

**为什么必要**（Ch12 血教训）：
- Ch12 Step 4 polish 修复 11 high + 3 medium，质量大幅提升
- 但 polish 改写时**反向引入** 2 个 forbidden_items 黑名单词：
  - "像有人把那只举起的手轻轻放下了"（"轻轻" 是 ai_cliche 黑名单）
  - "仿佛刚才那句话和糖纸是同一类东西"（"仿佛" 是 ai_cliche 黑名单）
- audit E3 low warn 标识但已通过，问题渗入正文
- **跨章数据**：Ch1-12 累积 ai_cliche **36 处**（Ch5 11 处 / Ch9 7 处 / Ch10 1 处 / Ch12 2 处），主要由 polish 引入

**根因**：polish 修复时只盯审查 issue，没回头检 forbidden_items。

**复扫流程**（Step 4 anti_ai_force_check 之前必跑）：

```bash
# 1. 重跑 post_draft_check（已含 H22 AI cliche + H23 破折号扫描）
python -X utf8 "${SCRIPTS_DIR}/post_draft_check.py" ${chapter_num} --project-root "${PROJECT_ROOT}"
# 任一 [AI_CLICHE] / [AI_CLICHE_TOTAL] / [DASH_DENSITY] block → polish 继续，不得进 Step 5

# 2. 对照 context 包 forbidden_items 全文复扫
python -X utf8 -c "
import json, pathlib, re
ctx = json.loads(pathlib.Path('${PROJECT_ROOT}/.webnovel/context/ch${chapter_padded}_context.json').read_text(encoding='utf-8'))
text = pathlib.Path('${PROJECT_ROOT}/正文/第${chapter_padded}章*.md').read_text(encoding='utf-8')  # 通配
for f in ctx['step_2a_write_prompt']['forbidden_items']:
    if isinstance(f,dict) and f.get('type')=='ai_cliche':
        for w in f.get('items',[]):
            n = text.count(w)
            if n: print(f'WARN polish 后仍含 ai_cliche {w}: {n} 次')
"
```

**硬规则**：
- post_draft_check 报 `[AI_CLICHE] 'X' N 次 ≥ block` → polish 继续修
- post_draft_check 报 `[AI_CLICHE_TOTAL] 累计 N 次 ≥ block 3.0/千字` → polish 大幅删除/换具象
- 单 word block 阈值参见 H22（按词性配置 · 默认 1.5/千字）
- 累计 block 3.0/千字 = 明显 AI 文风（必须降到 < 1.8/千字）
- 替换策略：用具体动作/感官细节替换副词（"轻轻按"→"按一下"/"拂过"；"仿佛 X"→直接比喻或 Show）

**项目级 override**：`.webnovel/ai_cliche_config.json` 可调阈值（如战斗章可放宽"猛地"）

## 2A. Anti-AI 检测细则（对应执行顺序第 3 步）

> 词频统计**仅作为提醒**，不再作为硬性门槛。若明显超标，需修复并简要说明。  
> 说明：即使词频仅作提醒，若最终文本仍有明显 AI 模板痕迹，`anti_ai_force_check` 仍应判定为 `fail`。

### 常见AI痕迹

| 特征 | 表现 |
|------|------|
| 句式规整 | 句子长度相近，结构相似 |
| 模式化词汇 | 首先、其次、最后、值得注意的是 |
| 过度连接 | 大量使用然而、因此、此外 |
| 情感平淡 | 缺乏个性化情绪表达 |
| 信息过密 | 每句都有信息，缺少留白 |

### 高频词提醒清单

| 类型 | 关键词 |
|------|--------|
| 总结词 | 综合 / 总之 / 由此可见 / 总而言之 |
| 列举结构 | 首先 / 其次 / 最后 / 第一 / 第二 / 第三 |
| 学术词 | 而言 / 某种程度上 / 本质上 |
| 因果连词 | 因为 / 所以 / 由于 / 因此 |

### 签名句式密度硬线（Round 17.2 · Ch8 P0-R5 根治 · 2026-04-24）

**引入背景**：Ch8 polish 后“没X” 34 次漏抓（editor_notes 目标 ≤15 · 旧 hygiene H21 阈值 30 太宽），用户手动 12 处改写才降到 10。polish-guide 只管 anti-AI 禁语，未管密度签名——必须补齐。

**项目级覆盖**：`.webnovel/signature_density_config.json` 可自定义阈值（字段名如 “没X”/“扶眼镜”）。

**实现层**：
- `scripts/post_draft_check.py` section 11 SIGNATURE_DENSITY 扫描（起草后闸门 · 超 block 阻断 Step 3）

### 第2层：句式规则（反模板）

- 禁止“首先/其次/最后”三段式说明写法。
- 禁止连续三句“主语 + 谓语 + 宾语”同构句。
- 禁止“他认为自己需要做三件事：一、二、三”清单化叙事。
- 每 3-5 段至少出现一次句长变化（短句打断/插入动作/对话反问）。

#### Round 19 Phase B 新增 4 句式（对应 RCA top 根因 N1/N2/N3/P4）

> 上轮 RCA §4 跨章重犯统计：5 类问题在 7-10 章 polish 反复修但都修不住。Phase B 把刻度量词/了一下/未字否定/系统术语百分比 4 条专项规则补入第 2 层（“不是X是Y” 已被 Phase A anti-ai-guide.md 起草前预防接住，不重复）。

| 句式 | 触发模式 | 阈值 | 严重度 | RCA 对应 |
|---|---|---|---|---|
| 刻度量词外溢 | 半度 / 半秒 / 半指 / 半分（任一组合）| 单章 ≥ 3 处 | block | N1（强信号必修）|
| “了一下”节拍密度 | “X了一下”句型（笑/愣/抖/顿/抽/眨/瞪 等）| ≥ 4 次/千字 | block | N2（强信号必修）|
| “未”字否定外溢 | “未+动词”（未答 / 未说 / 未应 / 未动 等）叙事/对话混用 | ≥ 2 处 | warn | N3（中信号必修）|
| 系统/RPG 术语百分比腔 | “X% 到 Y%” / “{N} 个刻度” / “稳态偏 X” | 单章 ≥ 1 处 | warn | P4（强信号必修）|

**执行规则**：

- 刻度量词外溢 / “了一下”密度命中 block → polish 必须降到 warn 以下，否则 anti_ai_force_check=fail
- “未”字否定外溢 / 系统术语百分比 warn → polish 阶段优先重写，剧情必要保留则记 deviation（位置 + 原因 + 代价）
- 与签名密度硬线表（“未X”/“笑了一下”等单词频次）共同执行；本规则是**句式结构维度**，签名密度是**词频维度**，互不替代

### 第3层：形容词与副词限制

- 连续两个以上形容词修饰同一名词，默认重写。
- 每 300 字中“很/非常/极其/格外/十分”总数建议不超过 4。
- 优先“动作 + 结果 + 代价”，弱化“抽象评价词”。

### 第4层：四字套语/成语限用

- 禁止高频空转四字词密集堆叠（如“心潮澎湃、百感交集、五味杂陈”连用）。
- 每 500 字四字套语建议不超过 3 个，且不得连续出现。
- 若使用成语，必须承担叙事功能（推进信息/冲突/角色态度）。

### 第5层：对话去AI化

- 禁止“说明书式对话”（完整解释背景、逻辑过满）。
- 对话必须有意图（试探/回避/施压/诱导/防御）。
- 允许真实口语停顿、打断、反问，避免全员“标准书面语”。

### 第6层：段落结构

- 单句成段建议占比 25%-45%（冲突章节可提高）。
- 段落长度建议 20-100 字，避免连续大段说明。
- 场景切换优先用动作/声音/位置变化，不用“第一幕/第二幕”标签。

### 第7层：标点节奏

- 禁止连续使用 `......` 或 `！！！`。
- 省略号、感叹号每段最多 1 次（特殊情绪爆点除外）。
- 长句中逗号超过 4 个时，优先拆句。

### Anti-AI 改写算法（执行动作）

1. 抽象情绪句 → `生理反应 + 当下意图 + 下一动作`
2. 结论句 → `事实细节 + 代价/风险 + 决策`
3. 连续说明句（>=3）→ 改成“对白/动作/反问”混排
4. 连续同构句（>=3）→ 至少打断 1 句为短句或插入动作锚点
5. 对话整段解释背景 → 改为“带意图的对抗式对白”

### 结构级 Anti-AI 检测（新增）

> 词频检测治标，结构检测治本。以下模式不依赖具体词汇，而是检测 AI 写作的深层结构特征。

1. **信息密度均匀度**: 真人写作信息分布有“疏密波动”，AI 偏“均匀撒布”
   - 检查：前/中/后三段的信息密度标准差
   - 标准差过低（< 阈值）= 疑似 AI 均匀分布模式
   - 修复：在低密度段增加细节，在高密度段留出呼吸

2. **情绪强度曲线**: 真人写作有“突然爆发”，AI 偏“平缓递进”
   - 检查：逐段标记情绪强度（1-5），检查曲线是否过于平滑
   - 全章情绪在 2-3 之间波动无爆发 = 疑似 AI 平缓模式
   - 修复：在冲突/高潮点制造情绪跳变

3. **段落功能分布**: 真人写作偶有“纯氛围/纯留白”段，AI 几乎每段都含“信息”
   - 检查：是否存在功能为“纯节奏/纯氛围”的段落
   - 全段落都是“信息+推进” = 疑似 AI 效率模式
   - 修复：适当插入纯氛围段（环境、声音、光影）

4. **句长方差**: 真人写作句长方差大，AI 偏均匀
   - 已由 prose-quality-checker 检测，此处做二次确认

### 通过阈值（手工判定）

- 全文不得保留未处理的三段式枚举模板句。
- 全文命中的高风险表达均已改写或记录 deviation。
- 至少完成 3 处“抽象句→行为句”改写（全文范围内统计）。
- 章末存在可感知的下一步压力或未闭合问题。
- 若仍保留高风险表达，需在 deviation 说明保留理由。

### Phase 1 终检（原版保留）

- [ ] 高风险词汇已做全文检查与必要替换。
- [ ] 未出现三段式说明句（首先/其次/最后）。
- [ ] 对话存在真实冲突意图，不是信息宣讲。
- [ ] No-Poison 毒点红线检查见第 5 节（本清单仅覆盖 Anti-AI）。
- [ ] 修改后未破坏“设定即物理 / 大纲即法律”。

## 3. 审查问题修复优先级

| 严重度 | 处理规则 |
|---|---|
| critical | 必须修复；无法修复必须记录 deviation 与原因 |
| high | 必须优先处理；无法修复记录 deviation |
| medium | 视篇幅和收益处理 |
| low | 可择优处理 |

类型对应修复动作：
- `OOC`：恢复角色话术、风险偏好、决策边界
- `POWER_CONFLICT` / `SETTING_CONFLICT`：能力回落到合法境界，或补出“获得路径+代价”
- `TIMELINE_ISSUE` / `CONTINUITY`：补足时间流逝锚点，修正事件先后顺序
- `LOCATION_ERROR`：补移动过程与空间锚点
- `PACING_IMBALANCE` / `PACING`：增加缺失的推进事件或删冗余说明段
- `CONTINUITY_BREAK`：补衔接句与过渡动作
- `READER_PULL`：强化钩子锚点、补充微兑现、增加悬念管理
- `STYLE`：按 Anti-AI 规则改写模板腔/说明腔/机械腔句式；引用载体不当时改为第三方载体（互联网梗→弹幕/评论/配角，角色引用应符合人设语音规则）
- `DIALOGUE_FLAT`：为每个角色补充独立语言标签（句式长度/口头禅/信息密度差异），参考角色卡语音规则
- `DIALOGUE_INFODUMP`：将“说明书对话”改为“带意图的对抗式对白”——角色说出信息必须同时服务于试探/施压/回避/诱导等意图
- `DIALOGUE_MONOLOGUE`：将长独白（>200字）打断为对话交互，插入其他角色的反应/打断/反问
- `PADDING`：删除或压缩无信息增量的段落，替换为推进剧情/角色/情绪的内容
- `REPETITION`：合并重复表达，保留信息量最大的一处，删除其余
- `PROSE_FLAT`：句式单调处插入短句爆发或长句舒展打破节奏；陈腐比喻替换为与本作设定相关的独特表达；缺失感官处补充非视觉感官（听觉/触觉/嗅觉）；弱动词替换为精确动词；抽象描写补具象数字锚点；引用生硬时删除“正如XX所言”等引导框架，化用意境融入描写/对话，整段引用压缩为 1-2 句化用（详见 `references/writing/classical-references.md`）
- `EMOTION_SHALLOW`：情感直述改为行为展示（生理反应+无意识动作+环境投射）；情感突变处补充梯度递进信号（1-2句过渡）；情感场景补充物理/生理锚点；跨章情感惯性断裂处补衔接句
- `READER_FLOW`：**读者视角卡点，按 description 里的 `[category:XXX]` 分类修复**（对冲作者脑子里清楚但读者看不出的盲区）：
  - `JUMP_LOGIC`（跳跃推理）：主角内心推断/顿悟过快 → 在顿悟前插入一句观察线索 + 一句内心过渡（“等等——”/“我忽然看见...”），给读者一个可追溯的推导路径
  - `MISSING_MOTIVE`（动机悬空）：主角主动行为无前置 → 在动作前加一句内心触发条件（“外婆说过...”/“昨晚那一幕让我...”）或外部刺激（某个动作看见某物）
  - `UNGROUNDED_TERM`（术语无锚）：首次名词无 inline 解释 → 首次出现时用 **3 段内** 的 inline 解释（人物台词/手记引用/主角内心备注），不要留到后面章节再解释
  - `ABRUPT_TRANSITION`（突兀转场）：场景/情绪切换无过渡 → 插入一句时空/动作过渡（“几小时后”/“雨停了的时候我已经到了 X”）
  - `VAGUE_REFERENCE`（指代模糊）：代词距离远/称呼切换多 → 重命名为全名或补一个指示性词（“桌上那只盒子”）
  - `RHYTHM_JOLT`（节奏抖动）：段长/POV 突变 → 在突变边界插入过渡段（从短碎句到长叙事：补一句描述性收束；从描述到动作：补一句动作前预备）
  - `META_BREAK`（叙事出戏）：文青腔/AI 腔 → 直接删除作者总结句；替换过度文青比喻为角色身份内的具象表达

### Issue Type 映射说明

内部 checker（Step 3）与外部审查（Step 3.5）使用不同粒度的 issue type。Step 4 修复时按以下映射统一处理：

| 外部审查 type | 对应内部 type | 修复动作参考 |

| `READER_PULL` | reader-pull-checker 的 HARD/SOFT 编码 | 按上方规则 |
| `STYLE` | Anti-AI 相关 | 按 §2A Anti-AI 规则 |
| `DIALOGUE_FLAT` | dialogue-checker 同名 | 按上方规则 |
| `DIALOGUE_INFODUMP` | dialogue-checker 同名 | 按上方规则 |
| `DIALOGUE_MONOLOGUE` | dialogue-checker 同名 | 按上方规则 |

### reader_flow vs reader_pull 优先级（双层避重）

Step 4 润色遇到同一位置 `READER_FLOW` 和 `READER_PULL` 两个 issue 时：
- **reader_flow 代表读者实际体验**（能否读懂），优先修复 — 修完 reader_flow 通常会顺带解决 reader_pull 的钩子弱问题
- **reader_pull 代表作者工艺指标**（钩子/微兑现），若 reader_flow 已修且 reader_pull 仍有问题，独立补一次
- **不得重复修同一 quote**——先判相似度，若相同位置同一原因，按 reader_flow 的修复方向为准

## 3.1 六个跨章 Root Cause · 系统性修复规则（2026-04-13 新增）

> 基于 Ch1-5 审计发现的跨章共性问题根治。Step 4 润色时按此 6 条检查每章。这 6 条是**读者视角爆款门槛**——任何一条不达标都会让读者出戏弃读。

### RC1 · 金手指 vs 推理的 scope 边界（JUMP_LOGIC 根治）

**禁止模式**：
- “我一看见，就懂了” / “看见它的瞬间就懂” / “原来 X 是 Y”（作为唯一推理步骤）
- 主角从单个证据跳到完整结论，中间无查证/对照/试错

**允许模式**：
- **金手指触发**合法：“字/符号 → 直觉懂字义”（这是超自然设定）
- **推理/技能使用**必须有**外部锚点**：
  - 翻手记对照规则
  - 查签到簿 / 翻资料
  - 试错一次（第一次研朱砂太重、第二字走形）
  - 对照前案
  - 问他人确认
- **升级顿悟**前必须有**铺垫层**（2-3 段具体事件 → 主角内心提炼 → 顿悟句）

**修复算法**：
1. grep “我.{0,5}懂了” / “就懂了” / “原来.{0,10}是” — 每个命中位置判断是金手指还是推理
2. 若是推理 → 前段插入 1-3 段外部锚点动作
3. 若是升级顿悟 → 确认前有 2-3 段铺垫，否则补

### RC2 · 章末收束范式（META_BREAK 根治）

**禁止模式**：
- 作者总结腔：“从今晚开始 / 从此 / 从那之后，X 再也不是 Y”
- 主题金句宣告：“原来 X 是 Y，Z 不是 W”（作为章末独立段）
- 文艺氛围收束：“外面的雨，忽然停了” 单独成段作章末

**允许模式（章末三件套）**：
1. **具体动作**（主角正在做某事，动作未完）
2. **具体画面**（环境细节/感官锚点）
3. **悬念锚点**（具体人/物/事/线索，非抽象概念）

**示例对比**：
- ❌ 差：“从今晚开始，我看见的东西，再也不是这个世界原本让我看见的东西了。”
- ✅ 好：“我低头看手心。手心什么都没有。窗外的雨打在铁皮棚上，一下，一下。手机在外套口袋里震了一下——陌生号码。”

**修复算法**：
1. 读最后 200 字，检查是否含作者总结腔关键词：“从今晚开始/从此/再也不是/第一次真的/原来.{0,8}不是”
2. 命中 → 删除整句，替换为“动作 + 画面 + 悬念锚点”三件套
3. 章末禁止独立抽象句成段

### RC3 · 新术语首次 inline 解释（UNGROUNDED_TERM 根治）

**触发条件**：
- 术语首次出现（全书/全卷第一次）
- 术语非常识（例：“阳眼”/“第三十八任”/“未闭之言”/“归途未尽”/“镜匣” 非常识；“走/镜子/账册” 是常识字不需额外解释）
- 术语是**规则判决依据**（主角基于它做重要决定）

**硬约束**：
- 首次出现后 **3 段内** 必须有 inline 解释，形式任选一：
  - **手记原句直引**：“手记里写：'阳眼是看阴中字的那只眼。'”
  - **角色台词**：“老陈说过，第三十七任叫周长海。第三十六任叫周长河。一代一代，同姓不同命。”
  - **主角内心联想**：“第三十八任——前面还有三十七个人像我一样，坐在过这把椅子、翻过这本手记。”
  - **具象化对照**：“未闭之言。就是死的时候没说完的那句话。”

**特别：文言/古诗引入**
- 两句白居易《梦微之》之类的诗引**必须**在引用前后加明确标记：
  - 前：“我想起外婆念过的那首诗——”
  - 后：“君埋泉下泥销骨。我寄人间雪满头。（白居易，外婆教过我）”
  - 或用引号 + 出处。禁止裸引。

**修复算法**：
1. 每章末识别所有首次出现的术语/古诗（通过 `data-agent` 的 new_entities 追踪）
2. 对每个 ungrounded 术语，回到首次出现位置检查 ±3 段是否有 inline 解释
3. 缺失 → 补 1-2 句（首选手记原句）

### RC4 · 关键反应前置外部触发（MISSING_MOTIVE 根治）

**硬约束**：
- 每个**角色态度突变**（强硬→软化/冷漠→关心/敌对→道歉）前，必须有**可见外部触发**：
  - 他们看见了某个动作（主角盖住镜子 → 表姨眼神闪动 → 才软化）
  - 他们听见了某句话
  - 某个物件触动他们（老人照片让儿子膝盖弯了）
- 每个**主角行为**（主动动作）前必须有：
  - 触发刺激（看见/听见/回忆）
  - 或已建立的规则（外婆说过 / 手记规定）

**判准**：读者能在本章或前章找到“凭什么这么反应”的具体线索。

**修复算法**：
1. 每章识别 3-5 个关键反应（技能用/情绪爆发/态度转变）
2. 逆向检查每个反应前 30 段内是否有外部锚点
3. 缺失 → 补一句具体动作/眼神/物件描写

### RC5 · 跨章时间戳严格算术（ABRUPT_TRANSITION + 时间悖论根治）

**Step 1 context-agent 硬约束**：
- 每章 Contract 必填字段：
  - `prev_chapter_end_timestamp`（上章末发生时间，精确到小时）
  - `current_chapter_start_timestamp`（本章起始时间）
  - `time_gap_strategy`（若 > 30 分钟 gap，说明填充策略：省略 / 过渡段 / 回忆锚）
- 跨章时间戳**算术校验**：本章起点 ≥ 上章末点；若早于上章末点 → `CRITICAL fail`，章纲需重做

**Step 2A 起草约束**：
- 章首第一段必须含**时间 + 地点**锚点（不是“雨停了三小时”这种抽象时间）
- 示例：
  - ❌ “外面的雨整整停了三个小时。” → 读者不知道现在几点几日
  - ✅ “下半夜的雨停了一阵，又下起来。四月十六日凌晨两点，我从出租屋骑电动车...”

**continuity-checker 加项**：
- 新增“跨章时间算术”专项检查：读上章末 + 本章首，判断时间先后 + gap 填充

### RC6 · 指代明确化（VAGUE_REFERENCE 根治）

**硬约束**：
- 代词（“那东西/这/它/她”）距先行词（所指具体名词）**≤ 3 段**
- 超出 → 必须重命名或加指示短语（“桌上那只旧盒子”）
- 独立成段的抽象词（“过了。” / “糊了。”）必须在前一段有明确所指锚

**修复算法**：
1. grep 单字/双字段落（如独立的“过了。” / “糊了。” / “她。”）
2. 检查前 3 段是否有明确锚
3. 缺失 → 扩展为完整句（“零点过了，那个会让人回头的时间窗也过了。”）

## 4. 网文化分层规则

### Hard（必须满足）

1. 本章至少有 1 个明确推进点（信息/行动/关系/局势至少一项）。
2. 关键对话可判定意图（试探/施压/回避/诱导至少一种）。
3. 抽象判断句改为“动作/反应/代价”表达。

### Soft（建议满足）

1. 章首建议在前 200-400 字进入冲突/风险/强情绪。
2. 章节中段有节奏脉冲（参考 800-1400 字一波，短章至少一次）。
3. 后段或章末保留未闭合问题或下一步期待锚点。

### Style（按章型择优）

1. 避免连续大段纯解释，优先“信息+动作/反应”交替。
2. 允许平缓收尾，但不能机械截断。

## 5. No-Poison 毒点规避（5类）

1. 降智推进：角色忽略常识仅为推进剧情服务。
2. 强行误会：可一句话说清却长期拖延。
3. 圣母无代价：无边界原谅高风险对象。
4. 工具人配角：只在功能节点出现，没有独立动机。
5. 双标裁决：同类行为评价标准不一致且无叙事解释。

命中任一毒点时，必须补“动机/阻力/代价”中的至少两项。

## 6. 润色红线（不可突破）

- 不改剧情走向（大纲即法律）
- 不改设定物理边界（设定即物理）
- 不删除关键伏笔
- 不强行改写角色关系基线

## 7. 输出格式（润色完成后）

必须输出：
1. 润色后的章节正文
2. 修复摘要（结构如下）

```text
[润色报告]
- 严重问题已修复: N 处
- 高优先级已修复: N 处
- 中低优先级已修复: N 处
- Anti-AI 改写: N 处
- anti_ai_force_check: pass/fail
- 毒点风险: pass/fail
- 偏离记录:
  - {位置}: {原因}
```

若 `critical` 未清零，必须显式标注“未通过”，并返回 Step 4 继续修复。

## 7.5 润色迭代与收益递减判断（新增）

当 Step 4 需要多轮润色时（如首轮修复引入新问题），按以下规则判断是否继续：

| 轮次 | 条件 | 处理 |
|------|------|------|
| 第1轮 | 正常执行 | 修复所有 critical/high + Anti-AI + 毒点 |
| 第2轮 | 首轮修复后仍有 critical | 针对性修复，不重做全量检查 |
| 第3轮 | 连续2轮分数提升 < 3分 | 输出“收益递减警告”，建议停止润色 |

**收益递减判断**:
- 记录每轮润色后的预估分数变化
- 如果修复某 issue 导致其他维度预期下降 > 5分，标记“副作用风险”
- 允许标注“可接受的 medium”（不值得修的问题），避免过度打磨
- 最多 3 轮润色，超过 3 轮强制进入 Step 5

**输出附加字段**:
```text

- 首轮分数变化: +{N}分
- 收益递减: {是/否}
- 副作用风险: {有/无}
```

## 8. 对话辨识度终检

遮住人名，检查每段对话是否能分辨说话者。参考项目 `设定集/` 中各角色卡的“语音规则”段落，验证对话风格是否符合设定。

通用检查标准：
- 每个有对话的角色，其说话风格是否与角色卡定义的语音规则一致
- 主角对话是否体现核心性格特征（如决策风格、信息处理方式、口头习惯）
- 配角对话是否有独立的语言标签（口头禅/句式偏好/信息密度差异/语气特征）
- 不同身份层级的角色（如上级/下属/对手）说话方式应有明显区分

如果连续 3 轮对话中 2 个角色的句式长度和语气相似度过高，标记为需改写。

## 9. 网文模板套路检测

命中以下任一条即改写或记录 deviation：
1. **天才降智**：主角明显有更好选择但为剧情需要选了蠢办法
2. **敌人排队**：反派轮流来送，上一个被打败下一个才出现
3. **圣母病**：主角无代价原谅高威胁敌人
4. **全员恋爱**：所有异性角色都对主角有好感
5. **信息垄断**：只有主角一个人能想到显而易见的解决方案
6. **情绪告白**：用大段内心独白替代行动展示情感
7. **说明书对话**：角色对话只为向读者传递设定信息

## 10. 具象化终检

- 抽象的“很多/很少/很快/很重” → 替换为具体数字
- “实力很强” → 对比参照（“三个D级猎人围了五天”）
- “时间紧迫” → 倒计时（“七十二小时内抵达”）
- 每千字至少 3 处具象数字锚点
- 禁止连续 2 句以上使用“很/非常/极其”等抽象程度词

## 11. 爽点密度补种（Step 4 专项）

当 Step 3 的 `high-point-checker` 评分 < 70 时，必须在本步骤执行爽点补种。

**过渡章例外**：若本章已由 Context Contract 判定为过渡章（`is_transition=true`），且评分 ≥ 60，则不触发补种。过渡章允许低爽点密度但不允许“整章无收获”——仍需至少 1 个微兑现（参见 `core-constraints.md` Soft 规则）。

**检查流程**:
1. 确认本章 high-point-checker 评分是否 < 70（过渡章阈值降至 < 60）
2. 若是，定位爽点最薄弱的段落区间（连续 800+ 字无任何满足感触发）
3. 在薄弱区间插入以下至少一种微爽点：
   - **信息揭示**：角色/读者获得关键新信息（“原来如此”感）
   - **小胜**：主角在小范围内获得优势（不必改变全局）
   - **认可**：主角获得他人认可/惊讶/佩服
   - **逆转**：小规模的预期翻转（不必是大反转）
   - **代价兑现**：前文铺垫的代价在此处产生回报
4. 补种时不得破坏大纲和设定约束（红线不可突破）

**补种约束**:
- 每次补种不超过 200 字增量（微调而非重写）
- 优先利用已有素材（本章已出现的角色/道具/冲突）
- 禁止无因果的“天降爽点”（必须有前文铺垫基础）
- 补种后需重新检查上下文连贯性
