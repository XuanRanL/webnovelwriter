# Context Contract Template

> **定位**：Context Contract 已内置于 Step 1（Context Agent 生成执行包时产出），不再作为独立 Step 存在。本文档提供 Contract 的字段 schema 与校验规则，供 context-agent 引用。

## 目标

把 `extract_chapter_context.py` 的上下文与 guidance 收束成”可执行合同”，直接驱动 Step 2A。

## 必填输出结构（Scene-Sequel 最小闭环）

- 目标（20字内）
- 阻力（20字内）
- 代价（20字内）
- 本章变化（30字内，优先可量化：关系/资源/风险/地位/能力）
- 未闭合问题（30字内，后段或章末）
- 核心冲突一句话
- 开头类型（冲突/悬疑/动作/对话/氛围）
- 情绪节奏（低→高/高→低/低→高→低/平稳）
- 信息密度（low/medium/high）
- 是否过渡章（true/false，必须按大纲判定，不按字数判定）
- 追读力设计（钩子类型/强度、微兑现清单、爽点模式）
- 爽点规划（必填）：
  - 类型（装逼打脸/扮猪吃虎/越级反杀/打脸权威/反派翻车/甜蜜超预期/迪化误解/身份掉马/微兑现）
  - 铺垫来源（前文哪个事件/冲突/设定在本章可以兑现）
  - 兑现方式（一句话描述爽点如何触发+读者获得什么情绪回报）
  - 若为纯铺垫/过渡章：必须规划至少 1 个微兑现（信息揭示/小胜/认可/资源获取）
- 时间约束（必填）：
  - 上章时间锚点（时间+时段，如"末世第3天 黄昏"）
  - 本章时间锚点（时间+时段）
  - 允许推进跨度（如"半天"、"次日"）
  - 时间过渡要求（跨夜/跨日时需补写过渡；大跨度需说明）
  - 倒计时状态（若有活跃倒计时，记录当前值与推进方向）
- 情感锚点规划（有情感场景的章节必填，纯过渡章可简化为惯性衔接）：
  - 情感场景识别：本章包含的情感场景类型（爆发/转折/积累/克制/共鸣），标注所在 beat
  - 锚点分配：每个情感场景至少指定 1 种锚点类型（生理反应/微表情/无意识行为/环境投射/感官变化）；重要情感高潮 ≥ 2 种
  - 梯度路径：高强度情感（爆发/崩溃/狂喜）标注 2-3 个递进信号（如 不安→焦虑→恐惧）
  - 跨章惯性：上章结束情绪 → 本章开头延续方式（承接/消化/转化）
  - Show:Tell 目标：全章 ≥ 2:1；重要情感场景 ≥ 3:1

过渡章判定规则（强制）：
- 依据章纲/卷纲中的章节功能标签与目标（铺垫/转场/承接/回收等）。
- 若大纲未显式标注，由“本章核心目标是否以推进主冲突为主”判定。
- 禁止使用字数阈值判定过渡章。

## 差异化检查

- 钩子类型优先避免与最近 3 章重复；连续 2 章相同即触发 warning，连续 3 章即 risk，连续 4 章即 critical（必须强制变更）。
- 开头类型优先避免与最近 3 章重复；连续 2 章相同即触发 warning，连续 3 章即 risk，连续 4 章即 critical（必须强制变更）。
- 爽点模式优先避免与最近 5 章重复。

若必须重复，必须记录 Override 理由，并至少变更以下一项：
- 对象
- 代价
- 结果
- 情绪（如同类型钩子但情绪基调不同）

## 题材快速调用（仅命中时）

命中题材：`esports` / `livestream` / `cosmic-horror`

执行：
1. 从 `writing/genre-hook-payoff-library.md` 选 1 条期待锚点（优先章末，也可后段）。
2. 选 1-2 条微兑现，优先与本章核心冲突同方向。

## 读取优先级

1. 必读：`writing_guidance.guidance_items`
2. 条件必读：`rag_assist`（`invoked=true` 且 `hits` 非空）
3. 选读：`reader_signal`、`genre_profile.reference_hints`

## 风险预估（新增）

在合同设计阶段，基于当前合同内容预测 Step 3 可能出现的高频问题，并在 Step 2A prompt 中预防性规避。

### 预测规则

| 合同特征 | 预测风险 | 预防措施 |
|----------|---------|---------|
| 章内目标 ≥ 4 个主要事件 | density-checker: 信息过密 / prose-quality-checker: 节奏单调 | 在 Step 2A 指令中强调"每个事件间留出呼吸段" |
| 无明确冲突设计 | reader-pull-checker: HARD-004 冲突真空 | 在合同中补充至少 1 个微冲突 |
| 情感高潮但无铺垫来源 | emotion-checker: 强行煽情 | 在合同中标注铺垫来源（前文哪个事件/关系） |
| 多角色对话场景 | dialogue-checker: 声音辨识度低 | 在 Step 2A 指令中列出每角色的语言标签 |
| 过渡章但无微兑现设计 | reader-pull-checker: 微兑现不足 | 在合同中规划至少 1 个微兑现 |
| 战斗/动作场景 | prose-quality-checker: 感官贫乏 | 在 Step 2A 指令中提示"补充听觉/触觉" |
| 情感场景但锚点规划为空 | emotion-checker: 锚点缺失/情感直述 | 在情感锚点规划中为每个情感场景指定锚点类型与梯度路径 |

### 输出字段

在合同 JSON 中增加：
```json
{
  "risk_prediction": {
    "predicted_weak_dimensions": ["信息密度", "情感表现"],
    "preventive_notes": ["事件间留呼吸段", "情感爆发需引用第X章铺垫"]
  }
}
```

## 读者认知追踪（新增）

追踪"读者当前知道什么"，服务于悬念管理和爽点设计。

### 必填字段

```json
{
  "reader_knowledge": {
    "reader_knows": ["主角真实身份", "反派的弱点", "宝物的位置"],
    "reader_expects": ["主角会在本章突破", "反派即将出手"],
    "info_gap": {
      "reader_knows_character_doesnt": ["反派已经盯上他"],
      "character_knows_reader_doesnt": ["主角暗中准备了后手"]
    }
  }
}
```

**用途**:
- 悬念钩设计：利用 info_gap 制造悬念
- 迪化误解设计：利用读者优越感
- 反转设计：打破 reader_expects
- 信息揭示型微兑现：从 character_knows_reader_doesnt 中选择
